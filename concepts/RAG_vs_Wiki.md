# RAG vs Wiki 对比

## 两种知识管理范式

### RAG (Retrieval-Augmented Generation)

**工作原理**:
1. 用户上传文档集合
2. 系统将文档分块并向量化
3. 查询时检索相关片段
4. LLM 基于检索片段生成答案

**特点**:
- ✅ 实现简单，开箱即用
- ✅ 支持大量文档
- ❌ 每次查询重新推理
- ❌ 无知识积累
- ❌ 复杂查询效率低（需综合多文档）

**代表工具**:
- NotebookLM
- ChatGPT 文件上传
- LangChain RAG

### LLM Wiki

**工作原理**:
1. 用户提供参考资料
2. LLM 读取并提取关键信息
3. LLM 增量更新持久化 wiki（结构化 markdown）
4. 查询时直接读取已编译的知识

**特点**:
- ✅ 知识持续积累
- ✅ 交叉引用自动维护
- ✅ 矛盾自动标记
- ✅ 复杂查询效率高（知识已综合）
- ❌ 需要更多 LLM 调用（构建时）
- ❌ 需要人工审核（确保质量）

**代表工具**:
- Karpathy 的 LLM Wiki 模式
- Obsidian + LLM Agent

## 对比表格

| 维度 | RAG | LLM Wiki |
|------|-----|----------|
| **知识存储** | 原始文档 + 向量索引 | 结构化 markdown wiki |
| **查询时处理** | 检索 + 重新推理 | 直接读取已编译知识 |
| **知识积累** | 无 | 有（持续更新） |
| **交叉引用** | 无 | 有（自动维护） |
| **矛盾处理** | 无 | 自动标记 |
| **复杂查询** | 效率低 | 效率高 |
| **实现复杂度** | 低 | 中 |
| **维护成本** | 低 | 中（需定期更新 wiki） |
| **适用场景** | 快速原型、简单查询 | 深度研究、长期知识管理 |

## 选择建议

### 用 RAG 当：

- 需要快速搭建原型
- 文档量巨大（>1000篇）
- 查询简单（单文档即可回答）
- 无长期知识管理需求

### 用 Wiki 当：

- 深度研究一个主题（数周/数月）
- 需要知识积累和演化
- 复杂查询需综合多源信息
- 追求高质量知识结构

## 混合方案

**最佳实践**:
1. **初期**: 用 RAG 快速检索
2. **中期**: LLM 定期（如每周）将 RAG 索引转换为 wiki 页面
3. **长期**: 主要依赖 wiki，RAG 作为补充
